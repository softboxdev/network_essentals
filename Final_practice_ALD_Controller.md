## Примерные ответы для проверки

### 1. Принципы проектирования доменной инфраструктуры

**Ключевые принципы:**
- **Принцип сегментации**: Разделение сетей по уровням защищенности и функциональному назначению
- **Принцип минимальных привилегий**: Предоставление пользователям только необходимых для работы прав доступа
- **Принцип глубокой эшелонированной защиты**: Многоуровневая система защиты (периметр, сеть, хост, приложение)
- **Принцип единого управления**: Централизованное управление политиками безопасности и доступом
- **Принцип соответствия требованиям**: Обеспечение выполнения требований 152-ФЗ, 187-ФЗ, приказов ФСТЭК

**Три основных аспекта:**
1. **Безопасность**: Соответствие требованиям регуляторов, внедрение СЗИ
2. **Масштабируемость**: Возможность расширения инфраструктуры без кардинальных изменений
3. **Управляемость**: Простота администрирования и мониторинга

### 2. Выбор технологий доменных служб

**Для систем с повышенными требованиями безопасности:**
- **ALD Pro на Astra Linux SE** - встроенные средства мандатного контроля, сертификация ФСТЭК
- **Обоснование**: Прямая поддержка российских СЗИ, соответствие требованиям к защите гостайны

**Для legacy-приложений, требующих совместимости с Windows:**
- **Samba DC** - полная совместимость с протоколами Windows, поддержка групповых политик
- **Обоснование**: Обеспечение работы унаследованных приложений без изменения кода

**Для стандартных бизнес-процессов:**
- **FreeIPA** - открытое решение, богатый функционал, поддержка стандартов
- **Обоснование**: Снижение затрат на лицензии, гибкость настройки

### 3. Роль СЗИ в доменной архитектуре

**Критически важные функции СЗИ:**
- **Мандатный контроль доступа**: Разграничение доступа на основе меток конфиденциальности
- **Аудит и мониторинг**: Фиксация всех событий безопасности
- **Защита от НСД**: Контроль целостности, обнаружение вторжений
- **Криптографическая защита**: Шифрование данных и каналов связи

**Интеграция с доменными службами:**
- Взаимодействие через PAM-модули при аутентификации
- Применение политик доступа на основе доменных групп
- Централизованное управление через доменные контроллеры

### 4. Схема доверительных отношений

**Оптимальная схема:**
```
ALD Pro (основной) ←→ FreeIPA (вспомогательный)
    ↓
Active Directory (легаси, одностороннее доверие)
```

**Типы доверия:**
- **ALD ↔ FreeIPA**: Двустороннее транзитивное доверие для seamless-аутентификации
- **ALD → AD**: Одностороннее доверие для миграции и временного доступа
- **FreeIPA → AD**: Одностороннее доверие для совместимости приложений

**Обоснование**: ALD Pro как центральный элемент управления безопасностью, поэтапный вывод AD из эксплуатации.

### 5. Стратегия миграции пользователей

**Этапы миграции:**
1. **Подготовительный**: Инвентаризация AD, классификация пользователей, проектирование структуры ALD
2. **Пилотный**: Миграция тестовой группы, отработка процедур, обучение пользователей
3. **Поэтапный**: Миграция по подразделениям с контролем качества
4. **Завершающий**: Отключение старых учетных записей, обновление зависимостей

**Потенциальные проблемы:**
- Несовместимость атрибутов пользователей
- Проблемы с унаследованными приложениями
- Сопротивление пользователей изменениям
- Временное снижение доступности сервисов

### 6. Управление доступом в гибридной среде

**Механизмы контроля доступа:**
- **Единые группы доступа**: Создание cross-domain групп с синхронизацией членства
- **Условный доступ на основе атрибутов**: Правила доступа в зависимости от домена источника
- **Ролевая модель**: Назначение ролей независимо от доменной принадлежности
- **Временные ограничения**: Ограничение прав для пользователей из legacy-доменов

### 7. Мандатный контроль доступа

**Принцип работы:**
- Каждому субъекту (пользователю) и объекту (файлу) назначается уровень целостности (УЦ)
- Правило доступа: субъект может читать объект если УЦ_субъекта ≥ УЦ_объекта
- Правило записи: субъект может писать в объект если УЦ_субъекта ≤ УЦ_объекта

**Назначение уровней:**
- Автоматически на основе должности и подразделения
- Динамически в зависимости от контекста доступа
- Постоянно для критичных ресурсов

### 8. Политика паролей и аутентификации

**Требования к паролям:**
- **Обычные пользователи**: Длина 8+ символов, сложность, история 5 паролей
- **Привилегированные пользователи**: Длина 12+ символов, многофакторная аутентификация
- **Внешние пользователи**: Временные пароли с обязательной сменой

**Дополнительные механизмы:**
- Smart-карты и токены для администраторов
- Биометрия для доступа к критичным системам
- OTP для удаленного доступа

### 9. Аудит и мониторинг

**Критичные события для аудита:**
- **Высокая критичность**: Попытки эскалации привилегий, доступ к системным файлам, изменения политик безопасности
- **Средняя критичность**: Неуспешные попытки входа, изменения в группах доступа, доступ к конфиденциальным данным
- **Низкая критичность**: Успешный вход/выход, доступ к обычным ресурсам

**Классификация событий:**
- **События аутентификации**: Все попытки входа, смены паролей
- **События доступа**: Доступ к файлам, сетевым ресурсам, приложениям
- **События изменений**: Модификация конфигураций, политик, прав доступа
- **События безопасности**: Срабатывание СЗИ, блокировки, нарушения политик
